Architecture

BigFix agent runs as a Solaris SMF service (BESClient). It communicates via HTTP/HTTPS through a relay hierarchy: Root Server → Relay(s) → Agent. Agents evaluate "Fixlet" relevance locally and report only when state changes. Default heartbeat is ~15 minutes.

Cross-Platform Abstraction Layer

BigFix built two proprietary DSLs on top of the C++ agent core:

Relevance Language
Type: Declarative query
  • Reads endpoint properties (OS version, installed packages, disk usage, etc.)
  • Same query works on all platforms — name of operating system returns "SunOS" on Solaris
  • On Solaris, translates to C calls: uname(), kstat_open(), getifaddrs(), pkg list parsing
ActionScript
Type: Imperative remediation
  • Executes remediation actions (install packages, run scripts, modify config)
  • Abstracts platform differences — same action template works on Windows/Linux/Solaris
  • On Solaris, translates to: pkg install (IPS), pkgadd (SVR4), sh -c for scripts

Compliance on Solaris

BigFix recently updated its DISA STIG Checklist for Solaris 11 to support ORACLE SOLARIS 11 X86, V3R2 and ORACLE SOLARIS 11 SPARC, V3R2 benchmarks (published June 2025). CIS benchmarks are also available. All compliance checking runs natively in the C++ agent using Relevance expressions — no JRE dependency.

Architecture — Linear Chain P2P

Tanium's patented linear chain architecture is its key differentiator. Unlike BigFix's hub-spoke relay model:

How It Works
Communication: Proprietary TCP protocol
  • Server sends a query to a few "leader" agents
  • Leaders forward to neighbors, who forward to theirs — forming a dynamic chain
  • Results aggregate back up the chain, with each node summarizing
  • Queries millions of endpoints in under 15 seconds
  • On Solaris, uses standard POSIX sockets (socket(), connect(), send(), recv())
Modular Client Extensions
Design: Single binary + loadable modules
  • Core agent handles communication, registration, basic queries
  • Each module (Discover, Asset, Comply, etc.) installs a Client Extension (CX)
  • Extensions are signed .so shared objects or script packages
  • Agent verifies digital signatures before loading any extension
  • Allows feature updates without replacing the core agent

Critical Limitation — JRE Dependency for Compliance: Tanium Comply on Solaris requires administrators to upload Oracle JRE 8 manually because Amazon Corretto (the bundled default for Windows/Linux/macOS) does not support Solaris or AIX. This adds an extra step and dependency that other platforms don't require. The Tanium Scan Engine uses this JRE to evaluate OVAL/SCAP content for CIS, DISA STIG, and custom definitions.

07 Implementation Recommendation

A phased approach to add Oracle Solaris agent support to ManageEngine Endpoint Central:

Phase 1: Foundation MVP — 3–4 Months
C99 agent, SMF service, HTTPS communication, hardware/software inventory (kstat, prtconf, pkg list, pkginfo), remote execution, Zone detection. Achieves parity with Ivanti.
Phase 2: Patch & Compliance — 3–4 Months
IPS SRU patching with Boot Environment rollback, CIS/STIG compliance via native C engine (no JRE dependency — beats Tanium), CVE vulnerability matching, software deployment. Matches BigFix.
Phase 3: Market Differentiation — 4–6 Months
DTrace integration (USDT probes), Zone lifecycle management, real-time FIM (port_associate), ZFS snapshot management, Solaris Audit/BSM for EDR. Features NO competitor has ever built.
